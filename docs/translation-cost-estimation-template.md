# 翻译成本预估模板（DeepSeek / 其他 LLM 通用）

这个模板用于估算“批量翻译”与“日常增量翻译”的 API 成本。  
你只需要替换参数区的数字与单价，即可快速得到区间预算。

---

## 1) 参数区（先填写）

> 建议先按“低/中/高”三档填写，避免单点估算偏差。

### A. 数据规模

- 产品数量 `N_product` = ____（示例：5）
- FAQ 数量 `N_faq` = ____（示例：8）
- 文章数量 `N_post` = ____（示例：6）
- 案例数量 `N_case` = ____（示例：0）

### B. 单条文本平均长度（中文源文，按字符数估算）

- 产品单条平均字符 `C_product` = ____（示例：1200，含 name/desc/specs/body 等）
- FAQ 单条平均字符 `C_faq` = ____（示例：500）
- 文章单条平均字符 `C_post` = ____（示例：1500）
- 案例单条平均字符 `C_case` = ____（示例：1000）

### C. 翻译语种数

- 目标语种数量 `L` = ____（示例：5，对应 en/es/pt/ar/ru）

### D. Token 估算系数（经验值）

- 输入 token 系数 `k_in`（token/字符）= ____（建议 0.55~0.75，默认 0.65）
- 输出 token 系数 `k_out`（token/字符）= ____（建议 0.55~0.85，默认 0.70）

> 说明：不同语言与提示词会影响系数。先用默认值，后续用真实账单回填修正。

### E. 单价（美元 / 百万 token）

- 输入单价 `P_in` = ____（请以官方价格页实时值为准）
- 输出单价 `P_out` = ____（请以官方价格页实时值为准）

参考：<https://api-docs.deepseek.com/quick_start/pricing/>

---

## 2) 计算公式（可复制到 Excel / 表格）

### 2.1 源文本总字符

`Chars_total = N_product*C_product + N_faq*C_faq + N_post*C_post + N_case*C_case`

### 2.2 输入/输出 token

- `Tokens_in = Chars_total * L * k_in`
- `Tokens_out = Chars_total * L * k_out`

### 2.3 成本（美元）

- `Cost_in_usd = (Tokens_in / 1,000,000) * P_in`
- `Cost_out_usd = (Tokens_out / 1,000,000) * P_out`
- `Cost_total_usd = Cost_in_usd + Cost_out_usd`

### 2.4 成本（人民币）

- `Cost_total_cny = Cost_total_usd * 汇率`

> 汇率可先按 7.2 估算。

## 3) 你的当前场景示例（可直接改）

按你最近一次数据量：

- `N_product=5`
- `N_faq=8`
- `N_post=6`
- `N_case=0`
- `L=5`
- 默认系数：`k_in=0.65`，`k_out=0.70`

先给一组“中位字符”示例：

- `C_product=1200`
- `C_faq=500`
- `C_post=1500`
- `C_case=1000`

则：

- `Chars_total = 5*1200 + 8*500 + 6*1500 + 0*1000 = 19,000`
- `Tokens_in = 19,000 * 5 * 0.65 = 61,750`
- `Tokens_out = 19,000 * 5 * 0.70 = 66,500`
- 合计 token 约 `128,250`

如果你把官方价格填进 `P_in / P_out`，立刻可得总费用。  
从 token 量看，这个体量通常属于“低成本”档。

---

## 4) 三档预算模板（推荐）

为了更稳妥，做低/中/高三档字符假设：

| 档位 | C_product | C_faq | C_post | 说明 |
|---|---:|---:|---:|---|
| 低 | 800 | 300 | 1000 | 文本较短、结构字段少 |
| 中 | 1200 | 500 | 1500 | 常见企业站内容 |
| 高 | 1800 | 800 | 2500 | 长文案 + 详情较多 |

对每档分别算 `Chars_total -> Tokens -> Cost`，最终给业务一个区间：

- 预估区间：`[Cost_low, Cost_high]`
- 建议预算：`Cost_mid * 1.2`（留 20% 缓冲）

---

## 5) 线上控制成本建议

- 批量补历史数据时，先跑小样本（例如 1 产品 + 2 FAQ + 1 文章）验证质量与单条 token。
- 只翻译“有变更”的内容（保留 `translationSourceHash` 机制）。
- 对长正文可分段翻译，避免一次请求超长导致重试成本增加。
- 记录每次任务的 `文档数 / token / 费用`，2~3 次后可得到很准的内部单价。
